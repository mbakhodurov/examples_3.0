# Тестирование с моками (mockery)

Та же функция `getWeatherAdvice()`, но вместо стаба используется мок, сгенерированный mockery v3. Мок позволяет настроить ожидаемые вызовы и проверить, что зависимость вызвана с правильными аргументами.

## Концепция

**Mock** vs **Stub**:

| | Stub ([пример 4](../4_unit_test_with_stubs/)) | Mock (этот пример) |
|---|---|---|
| Поведение | Захардкожено | Настраивается per-тест |
| Верификация вызовов | Нет | Да — аргументы, количество, порядок |
| Код | Пишется вручную | Генерируется mockery |
| Гибкость | Одинаковое поведение для всех тестов | Разное поведение в каждом кейсе |

Expecter API:
```go
mockClient.EXPECT().GetTemperature(city).Return(float32(25), nil)
```

## На что обратить внимание

| Что | Файл | Зачем |
|-----|------|-------|
| Конфигурация mockery | `.mockery.yaml` | `include-interface-regex: ".*Client"` — генерация моков для интерфейсов |
| Фабрика мока в тесте | `get_weather_advice_test.go` | `weatherCenterClientMock func(t *testing.T) WeatherCenterClient` — каждый кейс настраивает свой мок |
| Тест ошибки зависимости | там же, кейс "Город не найден" | Мок возвращает `ErrCityNotFound` — проверяем, что ошибка пробрасывается через `ErrorIs` |
| gofakeit для данных | там же | `gofakeit.City()` — случайный город, показывает, что конкретное значение не важно |
| Автоматическая верификация | `mocks/mock_weather_center_client.go` | mockery v3 регистрирует `t.Cleanup()` — невызванные ожидания приведут к провалу теста |

## Как запустить

```bash
task mocks:gen       # Генерация моков
task test            # Запуск тестов
```
