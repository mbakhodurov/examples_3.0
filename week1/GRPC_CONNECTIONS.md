# gRPC соединения: как это работает

## Архитектура HTTP/2

gRPC использует **HTTP/2** как транспорт. Главное преимущество — **мультиплексирование**.

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  HTTP/1.1 (REST)                    │  HTTP/2 (gRPC)                         │
├──────────────────────────────────────────────────────────────────────────────┤
│  Клиент ──► Сервер                  │  Клиент ══════════════► Сервер         │
│     │         │                     │     ║  Stream 1: GetUser()             │
│  [запрос 1]   │                     │     ║  Stream 2: CreateOrder()         │
│     │         │                     │     ║  Stream 3: GetUser()             │
│  [ответ 1]    │                     │     ║  Stream 4: ListProducts()        │
│     │         │                     │     ║  ... все через ОДНО соединение   │
│  [запрос 2]   │ ← ждём ответа       │     ╚══════════════════════════════════│
│     │         │                     │                                        │
│  1 запрос = 1 TCP соединение        │  Все запросы через 1 TCP соединение    │
│  (или keep-alive с очередью)        │  параллельно, без блокировки           │
└──────────────────────────────────────────────────────────────────────────────┘
```

### Как работают Streams

**ОДНО TCP-соединение** обслуживает **ВСЕ RPC-вызовы** клиента к серверу:

```
Клиент                                    Сервер
   │                                         │
   │══════════ TCP Connection ═══════════════│
   │                                         │
   ├─── Stream 1: GetUser(id=123) ──────────►│
   ├─── Stream 2: CreateOrder(data) ────────►│  ← параллельно!
   ├─── Stream 3: GetUser(id=456) ──────────►│
   │                                         │
   │◄────────── Stream 2: OrderCreated ──────┤
   │◄────────── Stream 1: User{...} ─────────┤  ← ответы в любом порядке
   │◄────────── Stream 3: User{...} ─────────┤
```

- Вызываешь `GetUser()` — создаётся Stream 1 внутри соединения
- Параллельно вызываешь `CreateOrder()` — создаётся Stream 2 в **ТОМ ЖЕ** соединении
- Ещё один `GetUser()` — Stream 3, и т.д.

Каждый Stream — логический канал со своим ID. Фреймы данных (headers, data, trailers) маркируются Stream ID и чередуются в любом порядке.

## Ключевые понятия

### GOAWAY

**GOAWAY** — HTTP/2 фрейм, который сервер отправляет клиенту:

> "Я закрываю это соединение. Заверши текущие запросы и переподключись."

После получения GOAWAY клиент:
1. **Завершает** все активные стримы (RPC-вызовы) на этом соединении
2. **НЕ создаёт** новые стримы на этом соединении
3. **Открывает новое** соединение для следующих вызовов

```
Сервер                          Клиент
   │                               │
   │──── GOAWAY ──────────────────►│  "Закрываюсь через 5 сек"
   │                               │
   │◄─── Stream 5: завершается ────│  Клиент доделывает активные RPC
   │◄─── Stream 7: завершается ────│
   │                               │
   │     [соединение закрыто]      │
   │                               │
   │◄══════ НОВОЕ соединение ══════│  Следующий вызов — новый коннект
```

### "Тёплое" vs "Холодное" соединение

| Тип | Что происходит | Время |
|-----|----------------|-------|
| **Холодное** | TCP handshake → TLS handshake → HTTP/2 preface → RPC | 50-150мс+ |
| **Тёплое** | Просто открываем новый Stream | ~0мс |

**Тёплое соединение** — уже установленное и готовое к работе. Клиенты держат соединения "тёплыми", отправляя keepalive ping'и даже без активных запросов.

### ENHANCE_YOUR_CALM

**HTTP/2 error code** (0xb), который сервер отправляет вместе с GOAWAY-фреймом:

> "Успокойся! Ты слишком агрессивен."

⚠️ Это НЕ gRPC status code (как `UNAVAILABLE` или `RESOURCE_EXHAUSTED`), а код на уровне HTTP/2 транспорта.

Используется когда клиент нарушает политику сервера (например, пингует слишком часто).
На стороне клиента это обычно проявляется как ошибка соединения, а не как gRPC-ошибка с кодом.

## Настройки сервера (ServerParameters)

```go
import "google.golang.org/grpc/keepalive"

grpc.NewServer(
    grpc.KeepaliveParams(keepalive.ServerParameters{
        MaxConnectionIdle:     15 * time.Minute,
        MaxConnectionAge:      30 * time.Minute,
        MaxConnectionAgeGrace: 5 * time.Second,
        Time:                  5 * time.Minute,
        Timeout:               1 * time.Second,
    }),
    grpc.KeepaliveEnforcementPolicy(keepalive.EnforcementPolicy{
        MinTime:             5 * time.Minute,
        PermitWithoutStream: true,
    }),
)
```

### MaxConnectionIdle

**Максимальное время простоя** соединения без активных RPC.

| | |
|---|---|
| **ПРОБЛЕМА** | Клиенты открывают соединение и "забывают" про него. Утечка ресурсов: файловые дескрипторы, память, слоты в connection pool. |
| **РЕШЕНИЕ** | Сервер отправляет GOAWAY и закрывает неактивные соединения. |
| **Рекомендация** | 15-30 минут |

### MaxConnectionAge

**Максимальное время жизни** любого соединения (даже активного).

| | |
|---|---|
| **ПРОБЛЕМА** | При добавлении нового инстанса сервера клиенты продолжают использовать старые соединения. Балансировщик (L4/DNS) видит соединение только при создании. Новый сервер простаивает, старые перегружены. |
| **РЕШЕНИЕ** | Принудительная ротация — сервер отправляет GOAWAY. Клиент переподключается → попадает на балансировщик → равномерное распределение. |
| **Рекомендация** | 30 минут - 1 час |

```
Было:                              После ротации:
┌─────────┐                        ┌─────────┐
│ Server1 │◄══ 100 клиентов        │ Server1 │◄══ 67 клиентов
└─────────┘                        └─────────┘
┌─────────┐                        ┌─────────┐
│ Server2 │◄══ 100 клиентов        │ Server2 │◄══ 67 клиентов
└─────────┘                        └─────────┘
┌─────────┐                        ┌─────────┐
│ Server3 │◄══ 0 клиентов (новый!) │ Server3 │◄══ 66 клиентов
└─────────┘                        └─────────┘
```

### MaxConnectionAgeGrace

**Дополнительное время** для завершения активных RPC после отправки GOAWAY.

| | |
|---|---|
| **ПРОБЛЕМА** | При закрытии по MaxConnectionAge активные запросы будут прерваны → ошибки на клиенте. |
| **РЕШЕНИЕ** | Сервер отправляет GOAWAY, но ждёт grace-период. Активные RPC завершаются, новые клиент не создаёт. После grace — принудительное закрытие. |
| **Рекомендация** | 5-10 секунд |

### Time

**Интервал отправки** keepalive ping'ов клиенту.

| | |
|---|---|
| **ПРОБЛЕМА** | TCP-соединение может стать "мёртвым" (клиент упал, сеть разорвалась, NAT-таблица протухла), но сервер не узнает — TCP keepalive имеет огромные таймауты. Ресурсы под "зомби"-соединение. |
| **РЕШЕНИЕ** | gRPC-уровневые ping'и (HTTP/2 PING frame). Нет ответа → соединение мёртво → закрываем. |
| **Рекомендация** | 1-5 минут |

### Timeout

**Время ожидания ответа** (pong) на keepalive ping.

| | |
|---|---|
| **ПРОБЛЕМА** | Сколько ждать ответа на ping? |
| **РЕШЕНИЕ** | Если pong не получен за Timeout — соединение закрывается. |
| **Рекомендация** | 1-20 секунд |

⚠️ Слишком маленький → ложные срабатывания при временных задержках.
⚠️ Слишком большой → долго держим мёртвые соединения.

### MinTime (EnforcementPolicy)

**Минимальный интервал** между ping'ами от клиента.

| | |
|---|---|
| **ПРОБЛЕМА** | Злонамеренные или неправильно настроенные клиенты пингуют слишком часто (каждую секунду) → DoS. |
| **РЕШЕНИЕ** | Сервер отслеживает частоту. При нарушении → GOAWAY + ENHANCE_YOUR_CALM. |
| **Рекомендация** | Равен или чуть меньше Time |

### PermitWithoutStream (EnforcementPolicy)

**Разрешать ping'и** когда нет активных RPC (Streams).

| | |
|---|---|
| **КОНТЕКСТ** | Клиент хочет держать соединение "тёплым" для мгновенного использования. Для этого пингует даже без активных вызовов: 1) проверить что соединение живо, 2) не дать NAT/firewall закрыть "неактивное", 3) избежать handshake при следующем RPC. |
| **true** | Разрешить "тёплые" соединения. Клиент может пинговать без активных RPC. |
| **false** | Экономия ресурсов: нет активности → пусть переподключается. |
| **Рекомендация** | `true` для большинства случаев |

## Настройки клиента (ClientParameters)

```go
import "google.golang.org/grpc/keepalive"

conn, err := grpc.NewClient(
    "localhost:50051",
    grpc.WithTransportCredentials(insecure.NewCredentials()),
    grpc.WithKeepaliveParams(keepalive.ClientParameters{
        Time:                10 * time.Second,
        Timeout:             3 * time.Second,
        PermitWithoutStream: true,
    }),
)
```

### Time (клиент)

**Интервал отправки** keepalive ping'ов серверу.

| | |
|---|---|
| **ПРОБЛЕМА** | Клиент не знает, что соединение "умерло" (сервер упал, сеть разорвалась). Следующий RPC зависнет надолго, ожидая таймаута. |
| **РЕШЕНИЕ** | Клиент периодически пингует сервер. Нет ответа → переподключение. |
| **Рекомендация** | 10-30 секунд (чаще чем сервер, чтобы обнаружить проблему первым) |

⚠️ **Важно**: Time клиента должен быть ≥ MinTime сервера, иначе сервер закроет соединение с ENHANCE_YOUR_CALM.

### Timeout (клиент)

**Время ожидания ответа** на ping.

| | |
|---|---|
| **ПРОБЛЕМА** | Сколько ждать pong от сервера? |
| **РЕШЕНИЕ** | Нет ответа за Timeout → соединение считается мёртвым → переподключение. |
| **Рекомендация** | 1-5 секунд |

### PermitWithoutStream (клиент)

**Разрешать ping'и** когда нет активных RPC.

| | |
|---|---|
| **true** | Держать соединение "тёплым". Пинговать даже без активных вызовов. Первый RPC будет мгновенным. |
| **false** | Пинговать только при активных RPC. Соединение может быть закрыто сервером/NAT. Первый RPC после паузы — холодный старт. |
| **Рекомендация** | `true` если важна низкая латентность первого запроса |

## Connection Pool и балансировка

### Один клиент = одно соединение?

По умолчанию gRPC клиент держит **одно соединение** к каждому адресу. Но можно настроить пул.

```go
// Несколько соединений к одному серверу
conn, _ := grpc.NewClient(
    "localhost:50051",
    grpc.WithDefaultServiceConfig(`{
        "loadBalancingConfig": [{"round_robin":{}}]
    }`),
)
```

### Client-side балансировка

gRPC поддерживает **client-side load balancing**:

```
┌────────────────────────────────────────────────────────────┐
│                      gRPC Client                           │
│  ┌──────────────────────────────────────────────────────┐  │
│  │              Name Resolver (DNS/etcd/consul)         │  │
│  │         "myservice.local" → [ip1, ip2, ip3]          │  │
│  └──────────────────────────────────────────────────────┘  │
│                           │                                │
│  ┌──────────────────────────────────────────────────────┐  │
│  │              Load Balancer (round_robin)             │  │
│  └──────────────────────────────────────────────────────┘  │
│         │                 │                 │              │
│    ┌────┴────┐       ┌────┴────┐       ┌────┴────┐         │
│    │ Conn 1  │       │ Conn 2  │       │ Conn 3  │         │
│    └────┬────┘       └────┬────┘       └────┬────┘         │
└─────────┼────────────────┼────────────────┼────────────────┘
          │                │                │
          ▼                ▼                ▼
     ┌─────────┐      ┌─────────┐      ┌─────────┐
     │ Server1 │      │ Server2 │      │ Server3 │
     └─────────┘      └─────────┘      └─────────┘
```

Стратегии:
- `pick_first` — первый доступный (по умолчанию)
- `round_robin` — по кругу
- Custom — своя логика

## Retry и Hedging

### Автоматические retry

```go
serviceConfig := `{
    "methodConfig": [{
        "name": [{"service": "ufo.v1.UFOService"}],
        "retryPolicy": {
            "maxAttempts": 3,
            "initialBackoff": "0.1s",
            "maxBackoff": "1s",
            "backoffMultiplier": 2,
            "retryableStatusCodes": ["UNAVAILABLE", "RESOURCE_EXHAUSTED"]
        }
    }]
}`

conn, _ := grpc.NewClient(
    "localhost:50051",
    grpc.WithDefaultServiceConfig(serviceConfig),
)
```

### Hedging (спекулятивные запросы)

Отправляем запрос на несколько серверов одновременно, берём первый ответ:

```go
serviceConfig := `{
    "methodConfig": [{
        "name": [{"service": "ufo.v1.UFOService", "method": "Get"}],
        "hedgingPolicy": {
            "maxAttempts": 3,
            "hedgingDelay": "0.5s"
        }
    }]
}`
```

⚠️ **Только для идемпотентных запросов!**

## Таймауты и дедлайны

### Контекст с таймаутом

```go
// Таймаут на один вызов
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()

resp, err := client.Get(ctx, &GetRequest{Uuid: "123"})
```

### Дедлайн propagation

gRPC автоматически передаёт дедлайн между сервисами:

```
Client (deadline: 5s)
    │
    ▼
Service A (осталось 4.5s)
    │
    ▼
Service B (осталось 4.0s)
    │
    ▼
Service C (осталось 3.5s) ← если не успеет, вся цепочка получит DeadlineExceeded
```

### Рекомендации

| Параметр | Значение | Комментарий |
|----------|----------|-------------|
| Timeout на вызов | 1-30s | Зависит от операции |
| Keepalive Time (сервер) | 5 мин | Не слишком часто |
| Keepalive Time (клиент) | 10-30s | Чаще чем сервер |
| MaxConnectionAge | 30 мин | Для ротации |
| MaxConnectionIdle | 15 мин | Освобождение ресурсов |

## Диагностика

### Включение логов gRPC

```bash
export GRPC_GO_LOG_VERBOSITY_LEVEL=99
export GRPC_GO_LOG_SEVERITY_LEVEL=info
```

### Channelz (интроспекция)

```go
import "google.golang.org/grpc/channelz/service"

// Регистрируем channelz сервис
channelz.RegisterChannelzServiceToServer(grpcServer)
```

Потом можно запросить статистику через grpcurl:
```bash
grpcurl -plaintext localhost:50051 grpc.channelz.v1.Channelz/GetTopChannels
```

### Метрики

OpenTelemetry интеграция:
```go
import "go.opentelemetry.io/contrib/instrumentation/google.golang.org/grpc/otelgrpc"

conn, _ := grpc.NewClient(
    "localhost:50051",
    grpc.WithStatsHandler(otelgrpc.NewClientHandler()),
)
```
