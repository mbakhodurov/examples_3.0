# HTTP Server: таймауты и защита

## Зачем нужны таймауты?

HTTP-сервер без таймаутов уязвим к различным атакам и утечкам ресурсов.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         TIMELINE HTTP-ЗАПРОСА                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  TCP Accept     Headers received     Body received      Response sent       │
│      │                │                   │                   │             │
│      ├────────────────┤                   │                   │             │
│      │ ReadHeaderTimeout                  │                   │             │
│      │                                    │                   │             │
│      ├────────────────────────────────────┤                   │             │
│      │            ReadTimeout             │                   │             │
│      │                                    │                   │             │
│      │                ├───────────────────────────────────────┤             │
│      │                │           WriteTimeout                │             │
│      │                                                        │             │
│      │                                                        ├──────────►  │
│      │                                                        │ IdleTimeout │
│      │                                                        │ (keep-alive)│
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Атака Slowloris

Самая известная атака на HTTP-серверы без таймаутов.

### Суть атаки

HTTP-протокол: клиент отправляет заголовки, а **конец заголовков** обозначается пустой строкой `\r\n\r\n`.

```
GET / HTTP/1.1\r\n
Host: example.com\r\n
User-Agent: Mozilla/5.0\r\n
\r\n                         ← Вот это означает "заголовки закончились"
```

**Пока сервер не получит `\r\n\r\n`, он НЕ может обработать запрос** — он просто ждёт.

### Как работает атака

Атакующий:
1. Открывает соединение
2. Отправляет начало запроса
3. **Никогда не отправляет `\r\n\r\n`**
4. Периодически шлёт мусорные заголовки, чтобы соединение не закрылось по TCP keepalive

```
Атакующий                                  Сервер
    │                                         │
    ├─── "GET / HTTP/1.1\r\n" ───────────────►│
    │                                         │  Сервер выделил:
    │                                         │  - файловый дескриптор
    │                                         │  - горутину (в Go)
    │                                         │  - буфер для заголовков
    │                                         │  Ждёт \r\n\r\n...
    │    ... пауза 10 сек ...                 │
    ├─── "Host: example.com\r\n" ────────────►│
    │                                         │  Всё ещё ждёт \r\n\r\n...
    │    ... пауза 10 сек ...                 │
    ├─── "X-Garbage-1: aaaa\r\n" ────────────►│  ← Мусорный заголовок,
    │                                         │    чтобы соединение не умерло
    │    ... пауза 10 сек ...                 │
    ├─── "X-Garbage-2: bbbb\r\n" ────────────►│
    │                                         │
    │         (никогда не отправляет \r\n\r\n)│
    │                                         │
    ×1000 таких соединений = сервер исчерпал лимит
```

### Откуда берётся лимит?

**ОС ограничивает количество файловых дескрипторов** на процесс:

```bash
$ ulimit -n
1024          # Типичное значение по умолчанию
```

Каждое TCP-соединение = 1 файловый дескриптор.

- 1000 Slowloris-соединений = 1000 дескрипторов
- Новые клиенты получат ошибку: `too many open files`

Даже если поднять лимит до 65535, атакующий с 10 серверами может открыть 650К соединений.

### Почему это работает?

| Легитимный клиент | Slowloris |
|-------------------|-----------|
| Отправляет заголовки за 10-50мс | Отправляет 1 заголовок в 10 сек |
| Соединение занято 50мс | Соединение занято ЧАСАМИ |
| Освобождает ресурсы быстро | Держит ресурсы бесконечно |

### Последствия

- Сервер держит тысячи "полуоткрытых" соединений
- Каждое соединение занимает:
  - Файловый дескриптор
  - Память на буферы (~4-8 KB)
  - Горутину в Go (~2-8 KB стека)
- **Легитимные клиенты не могут подключиться** — `connection refused`

### Защита

```go
server := &http.Server{
    ReadHeaderTimeout: 5 * time.Second,  // ← Главная защита от Slowloris
}
```

Теперь: если за 5 секунд не получили `\r\n\r\n` — закрываем соединение.
Slowloris-атакующий больше не может держать соединение часами.

## Параметры http.Server

```go
server := &http.Server{
    Addr:              ":8080",
    Handler:           mux,
    ReadHeaderTimeout: 5 * time.Second,
    ReadTimeout:       15 * time.Second,
    WriteTimeout:      15 * time.Second,
    IdleTimeout:       60 * time.Second,
    MaxHeaderBytes:    1 << 20, // 1 MB
}
```

### ReadHeaderTimeout

**Максимальное время на чтение заголовков** запроса.

| | |
|---|---|
| **ПРОБЛЕМА** | Атака Slowloris: злоумышленник открывает много соединений и очень медленно отправляет заголовки, исчерпывая лимит соединений сервера. |
| **РЕШЕНИЕ** | Сервер закрывает соединение, если заголовки не получены за указанное время. |
| **Рекомендация** | 5-10 секунд |

⚠️ **Это КРИТИЧЕСКИ важный параметр!** Без него сервер уязвим к Slowloris.

### ReadTimeout

**Максимальное время на чтение ВСЕГО запроса** (заголовки + тело).

| | |
|---|---|
| **ПРОБЛЕМА** | Медленные клиенты долго отправляют большое тело запроса, занимая ресурсы сервера. |
| **РЕШЕНИЕ** | Отсчёт начинается с момента принятия соединения. Если запрос не прочитан полностью — соединение закрывается. |
| **Рекомендация** | 15-60 секунд для обычных API |

⚠️ **Осторожно с загрузкой файлов!** Если есть эндпоинты для загрузки больших файлов:
- Увеличьте это значение
- Или используйте отдельный сервер для загрузок
- Или обрабатывайте таймаут на уровне хендлера

### WriteTimeout

**Максимальное время на запись ответа** клиенту.

| | |
|---|---|
| **ПРОБЛЕМА** | Медленные клиенты долго читают ответ (или намеренно не читают), занимая ресурсы сервера. |
| **РЕШЕНИЕ** | Отсчёт начинается с момента окончания чтения заголовков запроса. Если ответ не записан — соединение закрывается. |
| **Рекомендация** | 15-60 секунд для обычных API |

⚠️ **Осторожно со стримингом!** Если есть:
- SSE (Server-Sent Events)
- WebSocket (после upgrade)
- Стриминг больших файлов

Для таких случаев:
- Используйте `0` (без таймаута)
- Или обрабатывайте таймауты на уровне хендлера
- Или выносите на отдельный сервер

### IdleTimeout

**Максимальное время ожидания следующего запроса** при keep-alive.

| | |
|---|---|
| **ПРОБЛЕМА** | После отправки ответа клиент может держать соединение открытым (keep-alive), но не отправлять новых запросов. Ресурсы под "спящие" соединения. |
| **РЕШЕНИЕ** | Сервер ждёт IdleTimeout для нового запроса. Если не приходит — закрывает соединение. |
| **Рекомендация** | 60-120 секунд |

```
Запрос 1    Ответ 1                    Запрос 2    Ответ 2
   │           │                          │           │
   ├───────────┤                          ├───────────┤
   │           │◄────── IdleTimeout ─────►│           │
   │           │     (ожидание запроса)   │           │
```

Если `IdleTimeout` не указан, используется `ReadTimeout`.

### MaxHeaderBytes

**Максимальный размер заголовков** запроса в байтах.

| | |
|---|---|
| **ПРОБЛЕМА** | Атакующий отправляет огромные заголовки (гигантские Cookie, тысячи кастомных заголовков), исчерпывая память сервера. |
| **РЕШЕНИЕ** | Сервер отклоняет запросы с заголовками больше лимита. |
| **Рекомендация** | 1-2 MB (по умолчанию 1 MB) |

```go
MaxHeaderBytes: 1 << 20,  // 1 MB
```

## Рекомендуемые значения

### Обычный API

```go
server := &http.Server{
    ReadHeaderTimeout: 5 * time.Second,
    ReadTimeout:       15 * time.Second,
    WriteTimeout:      15 * time.Second,
    IdleTimeout:       60 * time.Second,
}
```

### API с загрузкой файлов

```go
server := &http.Server{
    ReadHeaderTimeout: 5 * time.Second,
    ReadTimeout:       5 * time.Minute,   // Больше времени на загрузку
    WriteTimeout:      15 * time.Second,
    IdleTimeout:       60 * time.Second,
}
```

### API со стримингом (SSE)

```go
server := &http.Server{
    ReadHeaderTimeout: 5 * time.Second,
    ReadTimeout:       15 * time.Second,
    WriteTimeout:      0,                  // Без таймаута для стриминга
    IdleTimeout:       60 * time.Second,
}
```

Или управляйте таймаутом в хендлере:

```go
func sseHandler(w http.ResponseWriter, r *http.Request) {
    // Отключаем WriteTimeout для этого запроса
    rc := http.NewResponseController(w)
    rc.SetWriteDeadline(time.Time{})

    // ... SSE логика ...
}
```

## Graceful Shutdown

При остановке сервера нужно дать время на завершение активных запросов.

```go
server := &http.Server{...}

go func() {
    if err := server.ListenAndServe(); err != http.ErrServerClosed {
        log.Fatal(err)
    }
}()

// Ожидание сигнала остановки
quit := make(chan os.Signal, 1)
signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
<-quit

// Graceful shutdown с таймаутом
ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()

if err := server.Shutdown(ctx); err != nil {
    log.Printf("Forced shutdown: %v", err)
}
```

### Что делает Shutdown()

1. Перестаёт принимать новые соединения
2. Ждёт завершения активных запросов
3. Закрывает idle-соединения сразу
4. Если таймаут истёк — принудительно закрывает всё

## Дополнительные меры защиты

### Rate Limiting

```go
import "golang.org/x/time/rate"

limiter := rate.NewLimiter(100, 10) // 100 req/sec, burst 10

func rateLimitMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        if !limiter.Allow() {
            http.Error(w, "Too Many Requests", http.StatusTooManyRequests)
            return
        }
        next.ServeHTTP(w, r)
    })
}
```

### Request Size Limit

```go
func limitBodyMiddleware(next http.Handler, maxBytes int64) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        r.Body = http.MaxBytesReader(w, r.Body, maxBytes)
        next.ServeHTTP(w, r)
    })
}

// Использование: лимит 10 MB
mux := limitBodyMiddleware(router, 10<<20)
```

### Connection Limit

На уровне ОС или reverse proxy (nginx, envoy):

```nginx
# nginx
limit_conn_zone $binary_remote_addr zone=addr:10m;
limit_conn addr 10;  # max 10 соединений с одного IP
```

## Сравнение с gRPC

| Аспект | HTTP/1.1 | gRPC (HTTP/2) |
|--------|----------|---------------|
| Мультиплексирование | Нет (1 запрос = 1 соединение) | Да (много стримов в 1 соединении) |
| Keep-alive | IdleTimeout | MaxConnectionIdle |
| Защита от медленных клиентов | ReadTimeout, WriteTimeout | Keepalive ping/pong |
| Ротация соединений | Нет встроенной | MaxConnectionAge |
| Graceful shutdown | server.Shutdown() | server.GracefulStop() |
