# Workspace — multi-module монорепо

Два сервиса (gRPC + HTTP) с общими спецификациями в одном репозитории. Демонстрирует организацию multi-module проекта через Go Workspaces и replace-директивы.

## Концепция

В реальных проектах несколько сервисов часто живут в одном репозитории и делят общий код: proto-определения, OpenAPI-спеки, утилиты. Go Workspaces (с Go 1.18) позволяют работать с такими проектами без публикации модулей.

Ключевой вопрос — как связать модули между собой. Существует три подхода.

## Подходы к организации монорепо

### Replace-директивы (используется в этом примере)

Все модули всегда используют текущую версию кода (HEAD):

```go
// http/go.mod
require github.com/.../workspace/shared v0.0.0
replace github.com/.../workspace/shared => ../shared
```

`replace` — не временный хак, а штатное решение для монорепо. Относительные пути работают и в CI (клонируется весь репозиторий), и в Docker (копируются нужные директории).

Trade-offs:
- Простота: не нужно управлять версиями
- Атомарные коммиты: меняем библиотеку и потребителей вместе
- Жесткая связанность: breaking change в shared ломает все сервисы сразу

### Git Tags

Каждый модуль версионируется независимо:

```bash
git tag shared/v1.0.0
git tag http/v1.0.0
```

```go
// http/go.mod — БЕЗ replace
require github.com/.../shared v1.2.0
```

Trade-offs:
- Независимые релизы и версионирование
- Постепенное обновление зависимостей
- Больше работы при каждом релизе
- Performance penalty при большом количестве тегов

### Один модуль на весь репозиторий

Один `go.mod` в корне, все сервисы — пакеты внутри.

Trade-offs:
- Максимальная простота
- Нет изоляции зависимостей между сервисами

### Сравнение

| Подход | Версионирование | Связанность | Когда использовать |
|--------|----------------|-------------|-------------------|
| Replace | Нет (всегда HEAD) | Жесткая | Маленькая команда, быстрые итерации |
| Git Tags | Независимое | Слабая | Много команд, независимые релизы |
| Single Module | Общее | Средняя | Простота важнее гибкости |

## Роль go.work

`go.work` — опциональное удобство для локальной разработки, не обязательный компонент сборки:

| | `replace` в go.mod | `go.work` |
|-|-------------------|-----------|
| Коммитится в репо | да | обычно нет |
| Нужен для сборки | да | нет |
| Назначение | связь модулей | удобство IDE |
| Работает в CI | да | не используется |

## На что обратить внимание

| Что | Файл | Зачем |
|-----|------|-------|
| Workspace-определение | `go.work` | Список модулей для локальной разработки |
| Replace-директивы | `http/go.mod`, `grpc/go.mod` | Связь с shared-модулем через относительные пути |
| Общие proto | `shared/proto/ufo/v1/` | Один proto на оба сервиса |
| Общие OpenAPI | `shared/api/weather/v1/` | Одна спека на HTTP-сервис |
| Сгенерированный код в shared | `shared/pkg/` | proto + ogen — генерируется, не редактируется |

## Как запустить

```bash
task gen         # генерация всего кода (proto + ogen)
task run:grpc    # gRPC на :50051
task run:http    # HTTP на :8080 (в другом терминале)
task test:api    # API-тесты
```

## Структура проекта

```
workspace/
├── go.work                # workspace-определение
├── grpc/                  # gRPC-сервис (UFO)
│   ├── cmd/server/
│   └── go.mod
├── http/                  # HTTP-сервис (Weather)
│   ├── cmd/server/
│   ├── internal/handler/
│   └── go.mod
├── shared/                # общие компоненты
│   ├── proto/ufo/v1/      # proto-спеки
│   ├── api/weather/v1/    # OpenAPI-спеки
│   └── pkg/               # сгенерированный код
├── tests/
└── Taskfile.yaml
```
